1. getStoredResult disparaît. Le TLazyDir gère déjà le cache via LDefined. Si la clé est en LDefined, Skip retourne directement la valeur sans rappeler computeFor. Notre storedResults Map JS est un double cache inutile qui en plus crée une dépendance circulaire.

2. Plus d'invalidation, on SET. Quand les données arrivent, au lieu de invalidateKey (LDefined → LRemoved) puis attendre un re-pull qui rappelle computeFor, on écrit directement la nouvelle valeur dans le TLazyDir : LDefined(nouvelles données, reads). C'est un "push" direct, pas un "invalidate + lazy re-compute".

3. Transactions. L'écriture dans le TLazyDir doit passer par le pattern fork/merge comme CollectionWriter.update, pour ne pas avoir les races de transactions qu'on a eues.

4. fetch ne doit PAS attendre le service. La Promise retournée par fetch sert à une éventuelle initialisation, pas à attendre la réponse API. fetch doit : lancer l'appel async (fire-and-forget), retourner immédiatement. Quand la réponse arrive → appeler update avec la valeur (ou l'erreur).

5. Erreurs via update. Pas de callback error séparé (on l'a déjà viré). Mais en plus, même le .catch côté framework disparaît. L'erreur est gérée dans le fetch de l'example : callMeteoApi().then(value => update(value)).catch(err => update(errorValue)). Le type d'update accepte les erreurs.

6. L'example se restructure. MeteoService.fetch fait : appeler une méthode privée callMeteoApi(key) (async, fait le vrai appel réseau), .then(result => callbacks.update(result)), .catch(err => callbacks.update(errorResult)). Le fetch lui-même resolve immédiatement.
